## 🏗️ 最終架構

```
┌───────────────────────────────┐
│ cron-job.org（主要觸發）       │ ← 08:00 TW（外部 cron 服務、SLA 高）
│ Daily News Trigger job        │   GET /trigger?token=XXX
└─────────────┬─────────────────┘
              ↓ HTTP
┌───────────────────────────────┐
│ Cloudflare Worker             │ ← /trigger endpoint 收外部 cron
│ daily-news-line-bot           │   scheduled() handler 為 fallback
└─────────────┬─────────────────┘
              ↓ repository_dispatch
┌───────────────────────────────┐
│ GitHub Actions                │ ← 被 dispatch 觸發（主要）
│ daily.yml                     │   也保留 schedule cron 當第二層 fallback
└─────────────┬─────────────────┘
              ↓
     ┌─────────────────┐
     │  generate.py    │ (有 dedup guard + OpenAI retry)
     └──┬──────────┬───┘
        ↓          ↓
   OpenAI API    RSS (BBC, NYT)
        ↓
 ┌──────────────────────────────────┐
 │ /docs/                            │
 │  ├ YYYY-MM-DD.html + PNG          │
 │  ├ index.html (latest)            │
 │  ├ stories/YYYY-MM-DD.json        │
 │  └ stories/index.json (日期列表)   │
 │ → GitHub Pages 靜態 serve         │
 └──┬─────────────────────────┬──────┘
    ↓ broadcast/push          ↑ fetch
 LINE Messaging API           │
    ↓                     ┌───┴──────────────────────┐
 📱 使用者                 │ Cloudflare Worker         │
                          │ （同一個 Worker）         │
                          └───┬──────────────────────┘
                              ↕
                      LINE Webhook (follow/message 事件)
```

**規則**：
- **三層觸發保險**：cron-job.org（主）→ CF Worker cron（fallback 1）→ GitHub schedule（fallback 2）
- 任一觸發 → Worker `triggerDailyWorkflow()` → `repository_dispatch` → daily.yml → `broadcast` → 所有好友
- generate.py 有 dedup guard（今日 json 已存在就跳過），所以多重觸發不會重發
- 手動 workflow_dispatch → `push` API → 只有 Jason 收到（靠 `LINE_TEST_USER_ID` 環境變數判斷）
- **dispatch 失敗 → Worker 透過 LINE push 警報給 Jason**（靠 `JASON_USER_ID` 變數）
- Worker 從 GitHub Pages 抓 JSON（免費的只讀資料庫）
- 同一個 Worker 同時做四件事：(1) LINE webhook  (2) `/trigger` 接外部 cron  (3) `scheduled()` CF cron  (4) 失敗警報

## 🛠️ 常見修改場景

### 場景 1：想改 LINE 傳送的新聞風格（如更幽默、更正式）
改 `generate.py` 裡 Step 2 的 `rewrite_prompt` 文字 → `git commit && git push` → `gh workflow run daily.yml`（這次只推給 Jason 自己看）→ 滿意後下一次排程會用新 prompt。

### 場景 2：想加新的聊天指令（例如「隨機」回一則舊新聞）
改 `worker/src/index.js` 的 `handleTextMessage` → 新增一個正則判斷 → `cd worker && wrangler deploy`。不用改 GitHub repo。

### 場景 3：想改 Rich Menu 按鈕（加按鈕、改圖）
改 `worker/scripts/setup-richmenu.py` 的 `BUTTONS` 陣列（或佈局）→ 重跑：
```bash
cd worker/scripts
PYTHONIOENCODING=utf-8 LINE_CHANNEL_ACCESS_TOKEN="xxx" python setup-richmenu.py
```
腳本會自動刪舊選單、建新的、設為預設。

### 場景 4：想改排程時間
**改三個地方**（主觸發 + 兩層 fallback）：
1. **cron-job.org 後台**（主要觸發）→ Cronjobs → Daily News Trigger → Edit → 改時間（用 Asia/Taipei 時區，直接填當地時間）
2. `worker/wrangler.toml` 的 `[triggers] crons`（fallback 1，CF Worker cron）→ 改完 `cd worker && wrangler deploy`；UTC 時間
3. `.github/workflows/daily.yml` 的 `schedule.cron`（fallback 2，GitHub cron）→ commit push；UTC 時間
TW - 8 = UTC。

### 場景 5：想增加每日新聞數量（從 5 改成 7）
改 `generate.py` 的 `TOP_N = 5`。LINE Flex carousel 最多支援 12 張 bubble。

### 場景 6：想禁止某個排程日發（放假）
在 Actions 頁 → Daily News Comics → 右上 `···` → Disable workflow。或在 yml 加條件判斷（例如週末不發）。

## 📜 踩過的雷（避免重蹈）

1. **Gemini 免費圖像 API 其實不免費** — 網路傳聞 500/天免費是錯的，Google 實際免費層 = 0 張。改用 OpenAI `gpt-image-1`（$0.01/張）。

2. **GitHub Actions cron 延遲嚴重** — 免費層 cron 沒 SLA，實測延遲 **8+ 小時**（2026-04-22、04-23 都延遲到台灣下午 14:29 才跑）。**已改用 Cloudflare Worker cron 觸發 `repository_dispatch`**（秒級準時）。GitHub schedule 保留當 fallback，generate.py 用「今日 json 已存在就跳過」做 dedup 避免重發。

2a. **OpenAI `gpt-image-1` 偶發 500** — 2026-04-23 排程就是因為圖像 API 一次 500 整個爆掉。已加 retry helper（`openai_post_with_retry`），指數退避 5/10/20/40 秒共 4 次，covers chat + image 兩個 API 呼叫。

2b. **GitHub fine-grained PAT 預設 30 天會過期 + CF Workers free 層 cron 也會 miss/lag** — 2026-04-29 雙雷同時爆：04-23 設的 PAT 剛好那幾天到期 → Worker cron 跑了但發 dispatch 收 401 → 系統 fallback 到 GitHub schedule → 那條本來就會延遲 8 小時 → 14:42 TW 才送。**修法**：(1) PAT 重發、有效期改 1 年、權限改成 `Contents: Read and write`（不是 Actions！`repository_dispatch` API 看 Contents） (2) 加 `cron-job.org` 當主觸發（外部 cron 服務、SLA 比 CF/GitHub 高）+ Worker 加 `/trigger` endpoint（token 驗證） (3) Worker 加「dispatch 失敗 → LINE push 給 Jason」警報，下次 token 又過期當天就發現，不用等隔天看延遲。

3. **LINE 一個 bot 配兩個後台**：
   - `developers.line.biz` → API 金鑰、Webhook URL、Channel Secret
   - `manager.line.biz` → 自動回應、招呼訊息、Rich Menu 預覽、這類「機器人行為」
   - **如果看到「感謝您的訊息...本帳號無法個別回覆」那種罐頭訊息**，就是 OA Manager 的自動回應還沒關。

4. **Channel Secret ≠ Access Token ≠ Channel ID**：
   - Channel ID：公開數字
   - Channel Secret：32 字元 hash（驗簽章用）
   - Channel Access Token：200+ 字元（發訊息用）

5. **LINE 圖片 URL 必須 HTTPS 公開可存取** — 用 GitHub Pages serve。檔名不變 LINE 會快取舊圖，所以加 `RUN_STAMP` 時間戳。

6. **PowerShell / Python 中文路徑 + 中文輸出會崩** — 解法：`PYTHONIOENCODING=utf-8` 環境變數、避免中文路徑硬寫、console 設 UTF8。

7. **Windows curl (schannel) 有時跟 Cloudflare cert 不合** — 拿不到 HTTPS 回應時，改用 PowerShell 的 `Invoke-WebRequest` 測。

8. **LINE broadcast = 炸所有好友** — 手動測試要用 push API + 指定 userId，不然媽媽會一直被吵。靠 `LINE_TEST_USER_ID` 環境變數 + `github.event_name == 'workflow_dispatch'` 判斷做出來。

9. **wrangler OAuth 有時效** — 登入視窗打開後要幾分鐘內點完，不然會 timeout 拿新 URL。Token 存本機後一陣子會失效，重跑 `wrangler login`。

10. **Cloudflare Workers 第一次部署要先有 workers.dev 子網域** — 在 dashboard 建好才能 deploy。`fu1688888888.workers.dev` 已經建好不用再動。
